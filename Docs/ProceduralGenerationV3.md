# Procedural Generation V3

Это документация по архитектуре системы генерации корабля 3-го поколения (V3). В этой версии система расширяется для поддержки многоэтажных кораблей, интерактивных дверей, генерации лута и предоставляет глобальный API для интеграции с геймплейными системами (задачи, хаос, ИИ).

## 1. Анализ текущей архитектуры (V2)

### Существующие компоненты
*   **ShipGenerator:** Главный оркестратор. Работает как корутина на сервере. **Статус:** Оставить как есть, добавить вызовы новых этапов (лут, API).
*   **RoomGraph:** Строит логическое дерево. Сейчас строит "плоскую" структуру (1 этаж). **Статус:** Расширить для поддержки свойства `Floor` и переходов между ветками этажей.
*   **RoomPlacer:** Физически расставляет префабы с помощью DFS. **Статус:** Требует адаптации для сдвига по оси Y (высота этажа) и проверки вертикальных коллизий.
*   **DoorGenerator:** Просто спавнит префабы. **Статус:** Полная переработка. Должен спавнить `DoorController` (NetworkBehaviour) и связывать его с комнатами.
*   **StairGenerator:** Заготовка. **Статус:** Разработать логику соединения сокетов типа `Ceiling` и `Floor`.
*   **LayoutValidator:** Базовая проверка. **Статус:** Улучшить алгоритм поиска пути, чтобы учитывать этажи и лестницы.
*   **ShipTemplate / RoomDatabase / RoomDefinition:** Хранят статические данные префабов. **Статус:** Расширить настройки (списки лута, высота этажа, вероятность многоэтажности).
*   **RoomData:** Рудимент старой версии (V1). **Статус:** Переработать в `RoomInstance` — класс, хранящий runtime-состояние комнаты (питание, кислород, двери).

## 2. Архитектура Procedural Generation V3

### Диаграмма жизненного цикла генерации
1.  **Initialize:** Чтение `ShipTemplate`, установка сида.
2.  **Build Graph (RoomGraph):** Создание логических нод с указанием этажа (1, 2, 3...).
3.  **Place Rooms (RoomPlacer):** Размещение 3D-моделей по этажам (Y-offset).
4.  **Connect Vertical (StairGenerator):** Установка лестниц/лифтов между `Ceiling` и `Floor` сокетами.
5.  **Spawn Doors (DoorGenerator):** Создание `DoorController` и привязка их к `RoomInstance`.
6.  **Validate (LayoutValidator):** Проверка проходимости всего корабля (A*).
7.  **Spawn Systems:** Установка реактора, терминалов.
8.  **Spawn Loot (LootGenerator):** Разброс предметов на основе `RoomTags`.
9.  **Bind API (ShipManager):** Инициализация глобального кэша комнат.
10. **Teleport Players:** Перенос игроков на спавн.

### Новые и изменяемые классы
*   **`ShipManager` (Новый):** Синглтон/NetworkBehaviour для геймплея. Хранит глобальную карту корабля.
*   **`RoomInstance` (Новый/Замена RoomData):** Хранит `RoomDefinition`, текущий этаж, ссылки на прилегающие двери, состояние (горят ли фары, есть ли пробоина).
*   **`DoorController` (Новый):** Управляет створками, шлюзованием, реагирует на сеть и питание. Знает, какие две `RoomInstance` он соединяет.
*   **`LootGenerator` (Новый):** Читает `LootDatabase` и расставляет лут.

## 3. Подсистемы (Generation Pipeline)

### Multi Deck Generation (Многоэтажность)
*   **Хранение этажей:** Свойство `int Floor` в `RoomNode` и `RoomInstance`. Физически это смещение `transform.position.y` на `Settings.FloorHeight * (Floor - 1)`.
*   **Независимая генерация и слияние:** Этажи (Floor 1 и Floor 2) генерируются в графе параллельно и равномерно от общих узлов (например, Реактор), чтобы обеспечить максимальное перекрытие.
*   **Динамические лестницы (Dynamic Stairs):** В `RoomPlacer` лестницы больше не являются строгими нодами дерева. Вместо этого после размещения комнат алгоритм ищет идеально совпадающие дверные проемы между 1 и 2 этажом в 3D пространстве и "врезает" лестницы (`Room_Stairs`) для создания дополнительных коротких путей.
*   **Слияние дверей (Doorway Merging):** Если две большие смежные комнаты соединяются несколькими параллельными дверьми, `RoomPlacer` автоматически удаляет перегородку между ними и создает единый широкий дверной проем.

### Door System (Двери)
*   `DoorGenerator` находит пересекающиеся сокеты двух комнат.
*   Спавнит префаб интерактивной двери.
*   Назначает двери ссылку на `RoomInstance A` и `RoomInstance B`.
*   Если реактор выключается (Dark Ship), дверь переходит в заблокированный режим (открывается только ломом).

### Loot Generation (Генерация лута)
*   Выполняется после размещения всех комнат.
*   `LootGenerator` получает из `ShipManager` все комнаты с нужным тегом (например, `Storage`).
*   Проходит по списку `ItemSpawns` внутри `RoomDefinition`.
*   Спавнит случайные предметы из таблиц лута (Loot Tables), учитывая сложность корабля.

### Room Metadata (Метаданные комнат)
`RoomTags` (флаговый enum) становится главным источником истины.
Пример: Комната с тегами `Power | Engineering | DangerZone`.
Эти данные переносятся из `RoomDefinition` (ScriptableObject) в `RoomInstance` (Runtime) во время генерации.

### Gameplay Integration (API для геймплея)
`ShipManager` предоставляет публичные методы для других систем:
```csharp
public List<RoomInstance> GetRoomsByTag(RoomTags tag);
public RoomInstance GetRoomAtPosition(Vector3 position);
public List<DoorController> GetConnectedDoors(RoomInstance room);
```
Это позволит `TaskManager`'у легко найти идеальное место для поломки трубы:
`var targetRoom = ShipManager.Instance.GetRoomsByTag(RoomTags.Maintenance).GetRandom();`

## 4. Future Extensions
Архитектура позволяет в будущем добавлять:
1.  **Динамическое разрушение:** Комнаты смогут менять `RoomTags` на `Hazard` при пожаре.
2.  **Генерация внешнего корпуса:** Добавление сокетов типа `Window` или `Turret` на внешнюю обшивку.
3.  **ИИ Врагов:** Использование графа комнат для патрулирования (враг знает кратчайший путь от `RoomNode A` до `RoomNode B`).
